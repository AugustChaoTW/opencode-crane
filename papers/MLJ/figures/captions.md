## Figure 2: Overall System Architecture
This diagram illustrates the overall system architecture of the CRANE framework, emphasizing the separation of concerns and modular design. The system is divided into three primary layers: the User Layer, the CRANE Core System, and External & Local Resources. The User Layer handles interactions via IDE/Terminal and the Workflow Orchestrator, which automates multi-step processes. Central to the framework is the CRANE Core System, managed by the Model Context Protocol (MCP) Server. It routes data through the Execution Engine, comprising the Multi-Source Retrieval module (for fetching academic literature via external APIs like arXiv and Google Scholar) and the LLM Verification Pipeline (a robust 3-LLM consensus engine). All AI interactions are governed by the Protected Zones Manager, which strictly enforces boundaries between AI-modifiable components and immutable user resources. This modular architecture allows CRANE to perform autonomous research while strictly maintaining data integrity and user control over sensitive project artifacts.

## Figure 3: 3-LLM Verification Pipeline (Consensus Model)
The 3-LLM Verification Pipeline represents CRANE's consensus-driven mechanism for evaluating academic claims and literature. When a raw document or claim enters the pipeline, it is simultaneously processed by three parallel evaluators: GPT-4o (focusing on primary logic and structural integrity), Claude 3.5 Sonnet (assessing nuance, tone, and framing), and Gemini 1.5 Pro (specializing in rigorous fact-checking). Each model independently evaluates the input across three sequential verification stages: Relevance Check (scored 0-100), Quality Assessment (scored 0-100), and Citation Worthiness (a binary pass/fail decision). The individual outputs are aggregated by a Majority Voting & Scoring Engine, which determines the final consensus. Claims that pass this rigorous multi-model scrutiny are added to the verified knowledge base, while those failing to achieve consensus are rejected or flagged for human review. This triangulation mitigates individual model hallucinations and ensures high-fidelity academic output.

## Figure 4: Protected Zones & Workflow Integration
Figure 4 demonstrates the integration of CRANE's Protected Zones Architecture with dynamic workflow overlays, using the Journal Submission Pipeline as an example. The architecture enforces three distinct security levels: the Core Zone (immutable configurations, API keys, and model prompts), the User-Controlled Zone (manual edits, main LaTeX files, and critical experiment scripts), and the AI-Modifiable Zone (dynamically generated reports, bibliographies, and literature summaries). Overlaying this structure is the automated Journal Submission Pipeline, which interacts with these zones progressively. The pipeline initializes within the AI Zone, proceeds to the Coaching & Review stage (where the AI reads the User Zone to generate feedback, but modifications remain strictly manual), conducts Risk Assessment by measuring metrics within the User Zone, and finally generates artifacts like the cover letter back into the AI Zone. This layered approach ensures that workflow automation accelerates research processes without ever compromising the integrity of core user code or data.
